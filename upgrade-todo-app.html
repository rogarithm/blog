더 괜찮은 todo list를 만들려면 어떻게 해야할까?

todo list를 쓰다보면 '이건 어떻게 처리해야할까' 고민하다 흐지부지되는 todo item이 생긴다. 한 번에 할 수 있도록 적절히 일을 나눴다고 생각했지만, 막상 해보면 시간을 들였는데도 불구하고 일이 끝나지 않는 경우도 있고, 컨디션이 좋지 않아서 시간을 많이 썼음에도 불구하고 별다른 성과를 내지 못하는 경우도 있다.

할 일은 계속해서 생기기 마련이고, 그러면 전에 추가했지만 아직 끝내지 못한 일들이 새로 생긴 일과 합쳐져 어떤 걸 먼저 해야할지 선택해야 하게 된다. 그러고 나면 여기서 선택하지 않은 일은 언제 처리해야할지가 문제로 남게 된다.

할 일 관리와 관련한 이런 문제는 나를 오랫동안 괴롭혀왔다. 처음엔 의욕에 가득 차서 일거리를 잔뜩 생각해냈지만 점차 힘에 부쳐서 일부만 끝내고 나머지는 건드리지도 못한 채로 구석에 처박아두는 상황이 비일비재하게 벌어졌다.

이런 문제를 프로그램으로 풀 수 있지 않을까 했지만, 전까지의 내 깜냥으로는 감당하기 힘들었다. 그동안 여러 문제를 프로그램으로 어떻게 풀어낼지 고민하면서 전보단 실력이 괜찮아졌다는 생각이 들었고, 다시 한 번 어떻게 문제를 해결할 수 있을지 고민해보기로 했다.

어떻게 해야 crud를 더 편하게 할 수 있을까?

기본적인 crud를 구현할 줄 알게 되었지만, 이걸 실제로 쓴다고 하면 적잖이 불편할 것이다.

왜 불편할까? 다른 앱과 비교해서 생각해보자.

아이디어

스프링 웹 애플리케이션의 request 형식은 json이지만 이 형식으로 todo list를 작성하는 건 불편한 일이다.

처음부터 json으로 request를 만드는 대신, 일정 형식의 todo list를 텍스트 형식으로 쓰고, 웹 애플리케이션에 요청할 일이 생기면 todo form text->json 변환 레이어를 거쳐서 json을 만들어내고, 만들어진 json으로 애플리케이션에 요청하면 일반적으로 todo list를 쓰는 방식을 유지한 채 애플리케이션을 쓸 수 있어 더 편해질 것 같다.

예상되는 문제는 형식에 맞지 않는 todo list를 만들 경우 json으로의 변환이 제대로 되지 않아 애플리케이션에 요청 시 처리가 제대로 되지 않는 상황이다. 어쩌면 요청 전 단계에서 문제가 발생하도록 할 수도 있겠다. 이런 상황을 방지하기 위해, 1) git으로 todo list 변경 후 커밋할 때마다 todo list 형식이 올바른지 validation하는 방식, 2) 텍스트 에디터에서 todo list 수정 후 저장할 때마다 validation하는 방식이 떠올랐다. 첫 번째 아이디어의 경우, git을 써야만 validation을 하게 된다는 한계가 있다. 두 번째 아이디어의 경우, 에디터에서 저장하는 행위가 일어나는 시점을 알 수 있는지, 그리고 그 시점에 해당 파일의 텍스트 정보를 읽어와서 validation 작업을 할 수 있는지가 문제다. 모든 에디터가 이런 지원을 하지는 않을 것 같다. 따라서 애플리케이션 내에도 이러한 기능을 구현하는 것이 필요해 보인다.

어떤 todo list 형식이 내가 느끼는 불편함을 해결해줄만한 형식인지 잘 떠오르지 않는다.

기한이 있는 일이라면 날짜 정보를 todo item에 적을 수 있게 하고, 그 정보를 기반으로 알림을 보내줄 수 있다면 좋을 것 같다. 이때 날짜 정보는 년/월/일, 월/일 형식이나 x 일 뒤, x 시간 뒤 같은 식으로 입력할 수 있게 해서 이 정보를 써먹을 수 있게 만드는 걸 생각해봤다.

todo item 중 완료했거나 삭제할 것을 어떤 식으로 표현해야할까? 텍스트에선 완료했다면 해당되는 todo item을 지우거나 앞에 x를 표시하곤 한다. 하지만 웹 애플리케이션이 DB를 쓴다고 생각하면, 텍스트와 DB의 정보가 동기화되어야 할 테고, 무작정 텍스트를 지우는 게 가능하도록 만들면 처리가 아주 어려워질지도 모른다.
